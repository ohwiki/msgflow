# 文章抓取模式说明

## 两种处理模式

| 模式 | 速度 | 质量 | 适用场景 |
|------|------|------|---------|
| **快速模式**（Worker 内处理） | 秒级 | 一般 | 纯文字文章、新闻、随笔 |
| **精细模式**（GitHub Actions） | 约 30 秒 | 高 | 含代码块、技术教程、Prompt 分享 |

## 自动选择逻辑

系统默认根据文章内容自动选择：

- 检测到 `<pre>` 标签（代码块）→ 自动走精细模式
- 飞书文档 → 始终走精细模式（通过飞书 API 获取结构化数据）
- 其他 → 走快速模式

## 手动选择

抓取表单提供"强制精细处理"开关，勾选后无论内容是否有代码块都走 GitHub Actions。

**建议勾选的场景：**
- 文章排版复杂（多级标题、表格、嵌套列表）
- 包含 Prompt / 配置文件等需要保留格式的内容
- 快速模式结果不满意时重新抓取

## 两种模式的技术差异

### 快速模式（Worker）

```
URL → Worker 抓取 HTML → Turndown（JS）转 Markdown → 存入 D1/R2
```

- 优点：即时完成，无外部依赖
- 缺点：Turndown 对复杂 HTML 结构转换可能丢失格式

### 精细模式（GitHub Actions）

```
URL → Worker 抓取 HTML → 存入 R2 → 触发 Actions
    → Python 下载 HTML → BeautifulSoup 清洗 → markdownify 转换
    → 代码块 Prettier 格式化 → 回调 Worker 更新
    → 自动归档到 GitHub 仓库
```

- 优点：清洗质量高，代码块格式化，自动归档
- 缺点：需要 30 秒左右，消耗 GitHub Actions 分钟数

### 飞书专用模式

```
URL → Worker 记录 → 触发 Actions
    → Python 调用飞书 Open API → 获取 Block 结构化数据
    → 逐块转换为 Markdown（16 种 block type 精准映射）
    → 回调 + 归档
```

- 优点：质量最高，代码块语言标记、富文本样式完整保留
- 缺点：需要配置飞书应用凭据

## 费用

| 项目 | 免费额度 |
|------|---------|
| Worker 请求（快速模式） | 10 万次/天 |
| GitHub Actions（精细模式） | 2000 分钟/月（私有仓库） |
| R2 存储 | 10 GB/月 |
